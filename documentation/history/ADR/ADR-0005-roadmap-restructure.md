# ADR-0005 — Roadmap Restructure

**Project:** Living Systems Project (LSP)  
**Module:** HiveMind  
**Status:** Accepted  
**Date:** YYYY-MM-DD

---

# Context

Early HiveMind planning focused primarily on individual features and mechanics.

While this approach helped identify possible additions, it created several challenges:

- Features lacked clear dependencies.
- Foundational systems were mixed with optional ideas.
- Development priorities were difficult to determine.
- Long-term architecture was harder to maintain.

As the project vision developed, it became clear that HiveMind required a systems-first roadmap.

---

# Decision

The project roadmap was restructured around development phases rather than individual features.

The roadmap would prioritize:

1. Stable foundations.
2. Core system implementation.
3. World impact.
4. Theme and expansion.
5. Future experimental ideas.

---

# New Roadmap Structure

## Phase 1 — Foundation and Stability

Purpose:

Create the minimum required foundation for HiveMind.

Focus:

- Fabric setup
- Mod structure
- Entity registration
- Basic functionality
- Development environment

Status:

Complete

---

## Phase 2 — Drone Framework

Purpose:

Create the foundation for autonomous entities.

Focus:

- Drone architecture
- AI systems
- Behavior framework
- Entity relationships
- Hive preparation

Status:

Active

---

## Phase 3 — World Impact

Purpose:

Allow Hive systems to meaningfully affect the world.

Focus:

- Resource interaction
- Environmental systems
- Expansion mechanics
- Persistent world effects

---

## Phase 4 — Theme and Flavor

Purpose:

Develop the identity and experience of HiveMind.

Focus:

- Visual identity
- Lore
- Audio
- Player experiences
- Advanced interactions

---

## Future Concepts

Purpose:

Maintain a location for experimental ideas.

Examples:

- Advanced civilizations
- Additional life systems
- Large-scale simulations

These ideas should not distract from active development priorities.

---

# Alternatives Considered

## Feature-Based Roadmap

Rejected.

Reason:

A feature list does not properly represent system dependencies.

---

## Complete Specification Before Development

Rejected.

Reason:

The project requires iterative development and discovery.

---

## Open-Ended Development

Rejected.

Reason:

Without structure, priorities become unclear.

---

# Consequences

## Positive

- Clear development priorities.
- Better architecture planning.
- Easier progress tracking.
- Reduced feature creep.
- Better alignment with project philosophy.

---

## Negative

- Some interesting features may be delayed.
- Requires discipline to maintain focus.
- Requires updating phases as systems evolve.

---

# Long-Term Impact

The roadmap now reflects the project's development philosophy:

```
Foundation

↓

Systems

↓

Interaction

↓

Expansion

↓

Emergence
```

The project will prioritize creating strong foundations before adding complexity.
